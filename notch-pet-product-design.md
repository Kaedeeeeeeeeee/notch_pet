# Notch Pet — macOS 刘海たまごっち 产品设计文档

> **文档版本**: v1.0  
> **最后更新**: 2026-04-15  
> **阶段**: 概念设计 → MVP 规划

---

## 1. 产品概述

### 1.1 一句话描述

一款住在 MacBook 刘海里的像素风电子宠物，拥有たまごっち级别的养成深度、世代繁衍机制和轻量社交系统。

### 1.2 核心差异化

- **刘海原生体验**：宠物"住在"macOS 刘海位置，不是悬浮窗也不是 dock 图标，而是占据了一块通常被浪费的屏幕空间。用户余光就能感知宠物状态，形成"它一直在那里陪着我"的临场感。
- **たまごっち级养成深度**：不是桌面摆件，而是有完整生命周期、性格系统、进化分支、世代繁衍的养成游戏。
- **轻社交驱动的世代传承**：通过结婚配对机制，宠物的基因在用户之间流动，产生天然的社交破圈效应。

### 1.3 目标用户

25~35 岁、使用 Mac 的年轻成年人，具备以下特征：

- 对像素风、复古游戏美学有好感
- 喜欢可爱事物，有收集/养成欲
- 活跃于 Twitter/X、小红书、Product Hunt 等社区
- 对独立开发产品有天然好感

## 2. 核心机制

### 2.1 时间系统

与传统たまごっち最大的不同：**宠物的时间只在电脑活跃时流动。**

| 电脑状态 | 宠物时间 | 说明 |
|---------|---------|------|
| 正常使用中 | 正常流动 | 状态衰减、成长计时均正常运行 |
| 屏幕锁定 | 暂停 | 检测 `NSWorkspace` 的锁屏通知 |
| 系统休眠 | 暂停 | 检测 `NSWorkspace` 的休眠通知 |
| 关机 | 暂停 | 记录关机时间戳，重启后不补算衰减 |

**设计理由**：电脑不是便携游戏机，用户不应该因为出差一周回来发现宠物饿死了。宠物的时间轴绑定电脑活跃时间，消除了惩罚感，让养成变成纯粹的正向体验。

### 2.2 夜间睡眠机制

- 每天本地时间 21:00~22:00（可设置），宠物自动进入睡眠状态
- 睡眠期间所有状态冻结：不会饿、不会生病、不会降低心情
- 起床时间受性格影响（例如：懒惰性格赖床到 10:00）
- 时区跟随系统本地时间，不做额外处理

**设计理由**：赋予宠物"生活节奏"，让它感觉是一个有自己作息的小生命，而非纯粹的数值系统。同时避免用户深夜加班还要操心喂宠物。

### 2.3 状态系统

宠物拥有以下基础属性，随电脑活跃时间逐渐衰减：

| 属性 | 衰减速度 | 补充方式 | 过低后果 |
|------|---------|---------|---------|
| 饱腹度（おなか） | 中 | 喂食 | 生病 |
| 心情（きもち） | 慢 | 玩耍 | 性格向负面偏移 |
| 体力（げんき） | 快 | 休息/睡觉 | 行动变迟缓 |

## 3. 生命周期与世代系统

### 3.1 生命周期总览（10天/世代）

每一"天"的长度以电脑活跃时间计算，非自然日。

```
┌──────────────────────────────────────────────────────┐
│  Day 1-5: 幼年·成长期    Day 6-8: 成熟期    Day 9: 育儿日  Day 10: 告别日  │
│  ───────────────────    ──────────────     ─────────    ─────────   │
│  ● 外观逐步进化          ● 状态："可婚配"     ● 宝宝诞生     ● 父母 + 宝宝   │
│  ● 性格逐渐显现          ● 可分享配对二维码   ● 一家三口同框   共度最后一天   │
│  ● 不可配对              ● 扫码完成结婚       ● 遗传决定      ● 日终父母离开  │
│                         ● 配偶显示在房间中                   ● 宝宝入睡      │
│                                                             ● 新世代开始     │
└──────────────────────────────────────────────────────┘
```

### 3.2 阶段详细说明

**幼年·成长期（Day 1~5）**

- 宠物从蛋中孵化，外观为幼年形态
- 随着照顾方式不同，逐步进化出不同外观分支
- 性格参数在这个阶段被用户的行为"塑造"（喂食频率、玩耍偏好等影响性格倾向）
- 不可进行配对

**成熟期（Day 6~8）**

- 宠物到达最终形态，性格定型
- 状态切换为"可婚配"
- 用户可以生成配对二维码，分享给其他用户
- 配对成功后，配偶宠物出现在房间中
- 如果这三天内未完成配对，则视为"未婚"

**育儿日（Day 9）**

- 宝宝诞生，出现在房间中
- 一家三口同框展示
- 宝宝的外观和性格由遗传规则决定（见 3.3）

**告别日（Day 10）**

- 父母和宝宝共度最后一天
- 日终时播放告别动画
- 父母离开，宝宝入睡
- 次日宝宝作为新一代宠物醒来，新世代开始

### 3.3 遗传规则

**已婚宠物的后代：**

- 性格来自父方 → 外观来自母方
- 性格来自母方 → 外观来自父方
- 即：性格和外观做交叉遗传，不会同时来自同一方

**变异机制（低概率）：**

- 小概率出现父母双方都没有的稀有性格或外观特征
- 经过多个世代繁衍，偶尔冒出完全意想不到的宠物
- 为长期用户制造惊喜，也是社交传播的素材

**未婚宠物的后代：**

- 宠物离开时留下一颗完全随机的蛋
- 随机蛋不受任何遗传规则约束，物种和性格完全随机
- 因为没有遗传约束，反而有可能开出稀有组合（安慰奖机制）
- 叙事包装："虽然没有找到伴侣，但还是留下了生命的延续"

## 4. 性格系统

### 4.1 设计理念

参考《宝可梦》的性格系统，性格不仅是一个标签，而是实际影响宠物的行为、表情、动画和说话语气。

### 4.2 性格维度（示例）

| 性格 | 行为影响 | 动画特征 | 语气特征 |
|------|---------|---------|---------|
| 活泼 | 自发动作频率高 | 蹦跳多、移动快 | 元气、感叹号多 |
| 害羞 | 被点击时会后退一下 | 动作幅度小 | 犹豫、省略号多 |
| 高冷 | 喂食时偶尔拒绝 | 转头不看用户 | 简短、淡漠 |
| 贪吃 | 饱腹度衰减更快 | 经常看向食物方向 | 经常提到吃的 |
| 懒惰 | 起床时间晚 | 经常打哈欠 | 慢吞吞 |
| 暴躁 | 状态低时反应更激烈 | 跺脚、冒烟 | 大声、语气冲 |

### 4.3 性格的形成与固化

- 幼年期（Day 1~5）：性格处于可塑状态，受用户照顾方式影响
- 成熟期（Day 6+）：性格定型，不再改变
- 遗传时作为一个整体传递给后代

## 5. 社交配对系统

### 5.1 配对流程

```
用户 A（成熟期宠物）              用户 B（成熟期宠物）
      │                                │
      ├── 生成配对二维码 ──────────────→ 扫码
      │   （包含宠物ID + 一次性token）    │
      │                                ├── 本地验证对方宠物状态
      │                                │
      │    ←── 确认配对 ────────────────┤
      │                                │
      ├── 双方状态锁定为"已婚" ─────────┤
      │                                │
      ├── 交换宠物静态数据 ─────────────┤
      │  （外观、名字、性格、基础动画ID）  │
      │                                │
      ├── 配偶宠物出现在各自房间中 ──────┤
```

### 5.2 防重婚机制

- 核心原则：一只宠物在一个生命周期内只能结婚一次
- 配对成功后，双方状态不可逆地锁定为"已婚"
- 二维码中包含一次性配对 token，使用后失效
- **需要轻量线上验证**：通过 CloudKit 记录配对关系，每次扫码时查询对方宠物 ID 是否已在已婚列表中
- CloudKit 免费额度足以支撑这个量级的查询

### 5.3 配偶数据同步策略

- 扫码配对时一次性传输配偶宠物的静态数据
- 传输内容：外观数据、名字、性格参数、动画资源 ID
- 之后不再实时同步，各自本地独立渲染
- 动画资源随 app 本地内置，无需传输动画文件本身
- 双方不需要保持同步，只要信息对得上即可

## 6. 交互设计

### 6.1 两级交互层级

**第一层：刘海常驻（一瞥）**

- 所有阶段只显示用户自己的当前宠物，不显示配偶和孩子
- 宠物大小约 16×24 像素
- 通过轮廓和动画传达状态（开心蹦跳 / 饿了耷拉 / 生病抖动）
- 用户余光扫一眼就能感知宠物当前状况
- 鼠标悬停时宠物给出微妙反馈（抬头看用户），暗示可交互

**第二层：展开窗口（沉浸）**

- 触发方式：点击刘海区域（非悬停，避免误触）
- 窗口大小约 320×400
- 上半部分：像素风房间场景，宠物在房间中活动
  - 已婚后配偶也显示在房间中
  - Day 9~10 宝宝出现在房间中
- 下半部分：状态条 + 操作按钮
- 不同生命阶段展示不同 UI 元素：
  - 成熟期：出现"寻找配偶"入口 / 配对二维码生成按钮
  - 已婚后：显示配偶信息
  - Day 9~10：显示宝宝信息

### 6.2 反馈系统（替代系统通知）

不使用 macOS 系统通知，所有反馈通过 app 自身表达：

| 紧急度 | 表现形式 |
|-------|---------|
| 轻度 | 宠物表情变化、小幅度动画（左右张望、头上冒问号） |
| 中度 | 刘海区域窗口微震（`CGAffineTransform` 短促抖动） |
| 紧急 | 宠物哭泣动画 + 短促 8-bit 音效（几十毫秒提示音） |

**音效设计原则**：

- 统一 8-bit / chiptune 风格，与像素画风一致
- 极短极轻，不打断用户工作流
- 不能让用户觉得烦——宁可弱也不可过

## 7. 房间系统

### 7.1 基础房间

- 展开窗口时显示的像素风小房间
- MVP 阶段提供一个默认房间主题
- 房间是宠物、配偶、孩子共同生活的场景

### 7.2 房间装饰（后续扩展 & 内购）

- 可更换房间主题（和风、太空、森林、都市等）
- 可放置家具和装饰物
- 装饰品作为内购商品出售

## 8. 商业模式

### 8.1 定价策略：买断 + 内购

| 类型 | 内容 | 建议定价 |
|------|------|---------|
| 买断 | 完整养成体验（全部核心机制、世代系统、社交配对） | ¥500~800（$3.99~5.99） |
| 内购 - 装饰 | 衣服、帽子、配饰等穿戴物 | ¥100~200（$0.99~1.99）/件 |
| 内购 - 房间 | 房间主题包、家具套装 | ¥300~500（$2.99~3.99）/套 |
| 内购 - 季节限定 | 节日主题装饰包（圣诞、万圣节、樱花季等） | ¥300~500（$2.99~3.99）/套 |

### 8.2 设计红线

- **宠物本身永远不卖**：宠物多样性通过养成和社交遗传获得，不设付费购买宠物通道
- **不卖加速/复活**：不做"付费跳过痛苦"的设计，宠物的离开应该有真实的失去感
- **不卖功能，卖表达**：内购只涵盖装饰性内容，核心养成和社交体验完整包含在买断价中

## 9. 技术架构要点

### 9.1 技术选型

- **框架**: Swift + AppKit（原生 macOS 开发）
- **Menu Bar**: `NSStatusItem` 自定义 view，精确控制刘海旁渲染
- **展开窗口**: `NSPopover` 或自定义 `NSWindow`
- **数据持久化**: `UserDefaults` 或本地 JSON 文件
- **线上验证**: Apple CloudKit（配对防重婚验证）
- **动画**: `NSImage` 逐帧切换 或 Core Graphics 逐像素绘制

### 9.2 关键技术注意事项

- **App Store 沙盒兼容性**：需早期验证以下功能在沙盒环境下是否正常工作：
  - `NSWorkspace` 休眠/唤醒/锁屏通知监听
  - 窗口震动效果
  - CloudKit 访问
- **备选发行方案**：如沙盒限制过多，可同时通过官网直销非沙盒版本（Gumroad / LemonSqueezy）

### 9.3 数据结构预埋

即使 MVP 不实现全部功能，数据结构需为以下系统预留扩展性：

- 每只宠物从诞生起拥有全局唯一 ID
- 完整的成长记录（每日照顾行为日志）
- 性格参数向量
- 家谱数据（父方ID、母方ID、世代编号）
- 装饰品槽位
- 房间配置数据

## 10. 开发分期规划

### Phase 1 — MVP（核心养成循环）

- [ ] 刘海常驻像素宠物渲染 + 基础动画（idle、饥饿、开心、生病）
- [ ] 状态系统（饱腹度、心情、体力）及衰减逻辑
- [ ] 时间系统绑定电脑活跃状态（休眠/锁屏暂停）
- [ ] 夜间自动睡眠
- [ ] 点击展开 popover，显示房间场景 + 状态 + 操作按钮
- [ ] 性格系统（幼年期可塑、成熟期定型、影响动画和行为）
- [ ] 进化成长树（基础分支）
- [ ] 10 天生命周期完整流程
- [ ] 动画音效反馈系统（替代系统通知）
- [ ] 未婚离开 → 随机蛋 → 新世代

**MVP 验证目标**：这个体验到底有没有让人每天想打开电脑看一眼的吸引力？

### Phase 2 — 社交配对

- [ ] 配对二维码生成与扫码
- [ ] CloudKit 轻量验证（防重婚）
- [ ] 配偶数据交换与本地渲染
- [ ] 已婚状态房间展示
- [ ] 宝宝诞生 + 遗传规则
- [ ] 告别日动画
- [ ] 变异机制（稀有遗传）

### Phase 3 — 经济与自定义系统

- [ ] 积分 / 金币机制（照顾宠物获得货币）
- [ ] 衣服、配饰商店（内购）
- [ ] 房间主题与家具商店（内购）
- [ ] 穿戴系统 UI

### Phase 4 — 内容扩展与社交增强

- [ ] 更多宠物物种 + 进化分支
- [ ] 季节限定装饰包
- [ ] 生病系统 + 治疗机制
- [ ] 更丰富的性格种类
- [ ] 社交分享功能（截图/宠物卡片分享到社交媒体）
- [ ] 宠物图鉴 / 家谱查看器

## 附录 A：情感设计备忘

- 宠物离开的告别动画是整个产品最重要的情感时刻，需要精心设计
- "第一次经历宠物离开"是用户最有可能截图分享的节点
- 刘海上的一个小像素生物能带来的陪伴感，远超预期——把这件事做到极致
- 用户不是在玩一个游戏，而是在照顾一个住在电脑里的小生命
